Seamless Migration of Azure Cloud Services (classic) to Azure Cloud Services (extended support)

Azure Cloud Services is a widely used Platform as a Service (PaaS) offering that allows you to build, deploy, and manage scalable and highly available web applications and APIs. Recently, Microsoft introduced Azure Cloud Services (extended support), which provides several enhancements over the classic Cloud Services model.

This comprehensive guide will walk you through the process of migrating your existing Azure Cloud Services (classic) deployments to the newer Azure Cloud Services (extended support) platform. We’ll explore the key benefits, migration paths, and the step-by-step process to ensure a smooth and successful transition.

Key Benefits of Azure Cloud Services (extended support)

Azure Cloud Services (extended support) offers several advantages over the classic deployment model:

  1. Regional Resiliency: The extended support model provides regional resiliency, ensuring your applications can withstand data center failures and offer better availability.
  2. Feature Parity: Cloud Services (extended support) maintains feature parity with the classic deployment model, allowing you to seamlessly migrate your existing workloads.
  3. Azure Resource Manager Capabilities: The extended support model integrates with Azure Resource Manager, providing access to advanced features like role-based access control (RBAC), tags, policies, and deployment templates.
  4. Pricing Parity: Both the classic and extended support models offer similar pricing structures, ensuring a consistent cost experience.

Migration Paths: Redeploy vs. In-place Migration

Azure Cloud Services (extended support) supports two primary migration paths from the classic deployment model:

  1. Redeploy:

    • Customers can deploy a new cloud service directly in Azure Resource Manager and then delete the old cloud service in Azure Service Manager after thorough validation.
    • Redeploy allows you to define resource names, organize or reuse resources as preferred, and reuse service configuration and definition files with minimal changes.
    • You’ll need to orchestrate traffic to the new deployment and delete the old cloud services in Azure Resource Manager.
    • This approach offers more flexibility but requires additional time to migrate.

  2. In-place Migration:

    • The in-place migration tool enables a seamless, platform-orchestrated migration of existing Cloud Services (classic) deployments to Cloud Services (extended support).
    • For in-place migration, the platform defines resource names, organizes each deployment and related resources in individual Resource Groups, and modifies the existing configuration and definition files for Azure Resource Manager.
    • Migration retains IP addresses, and the data path remains the same. The platform also deletes the Cloud Services (classic) resources after migration.
    • This approach offers a quicker migration but with less flexibility.

When evaluating your migration strategy, consider your application’s requirements and future plans. If your application is not evolving, Cloud Services (extended support) may be a viable option as it provides a quick migration path. However, if your application requires a more modern feature set, you may want to explore other Azure services like Virtual Machine Scale Sets, App Service, Azure Kubernetes Service, or Azure Service Fabric.

Migration Steps

The migration process for Cloud Services (classic) to Cloud Services (extended support) involves the following four key steps:

  1. Validate Migration: This step checks if the migration will not be prevented by common unsupported scenarios.
  2. Prepare Migration: The platform duplicates the resource metadata in Azure Resource Manager, locks all resources for create/update/delete operations, and ensures the metadata is synchronized across Azure Service Manager and Azure Resource Manager.
  3. Abort Migration: If needed, this step removes the resource metadata from Azure Resource Manager and unlocks all resources for further operations.
  4. Commit Migration: This final step removes the resource metadata from Azure Service Manager, unlocks the resources, and completes the migration. After the commit operation, the abort option is no longer available.

Throughout the migration process, the Prepare, Abort, and Commit operations are idempotent, meaning you can retry them if they fail, and the platform will handle the recovery.

Supported Resources and Features

Azure Cloud Services (extended support) migration supports a wide range of resources and features, including:

  • Storage Accounts
  • Virtual Networks (excluding Azure Batch)
  • Network Security Groups
  • Reserved Public IP addresses
  • Endpoint Access Control Lists
  • User Defined Routes
  • Internal load balancer
  • Certificate migration to Key Vault
  • Plugins and Extensions (XML and JSON-based)
  • On Start / On Stop Tasks
  • Deployments with Accelerated Networking
  • Deployments using single or multiple roles
  • Basic load balancer
  • Input, Instance Input, and Internal Endpoints
  • Dynamic Public IP addresses
  • DNS Name
  • Network Traffic Rules

Additionally, the migration process supports various configuration scenarios, such as virtual networks containing Microsoft Entra Domain Services, cloud services with a single deployment slot, and deployments not in a publicly visible virtual network.

Conclusion

Migrating your Azure Cloud Services (classic) deployments to the Azure Cloud Services (extended support) model can provide you with significant benefits, including regional resiliency, feature parity, and integration with Azure Resource Manager. By understanding the migration paths, supported resources and configurations, and the step-by-step process, you can ensure a smooth and successful transition for your cloud-hosted applications and services.
